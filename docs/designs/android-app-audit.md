# Design Audit: Pimote Android App

> Generated by `ux-review`. Audited against the historical brief at `docs/designs/android-app.md` (recovered from commit `625c31b` — the file was deleted during cleanup but is the original direction the implementation aimed at). Scope: all three rendered surfaces (Setup, Contacts, In-Call) plus the shared theme/components layer.

## Verdict

**❌ NOT READY** — 3 CRITICAL, 7 HIGH, 6 MEDIUM, 2 LOW.

The visual system (palette, type, spacing, component shells) is faithful and well-implemented. The failures cluster in three places: (1) the In-Call screen — the user's most important and most eyes-off surface — has a wrong-state ring color and missing context; (2) the project-header treatment in Contacts ships with a sub-spec hit target on the only tappable element; (3) Settings has no return path back to Contacts. Each of those three is independently shipping-blocking for the audience the brief defines (developers driving / using Auto).

---

## CRITICAL

### In-Call screen → avatar ring color

- **Connecting-state ring uses the wrong semantic color.** `AvatarRing.kt:60–64` maps `AvatarRingState.Connecting → colors.active` (teal `#4DC896`). The brief specifies _"Stroke: 2dp ring in Indigo (call active), Danger (call ended with error), Idle (connecting states)"_. The implementation has effectively swapped two states' semantics: connecting now reads as "alive/connected" (teal), and only the static Active state reads as Indigo. For an eyes-off surface, this is a real misread — the user glances at the screen, sees teal-green and assumes they're through; in fact they're still mid-handshake.
  - **Fix:** Connecting ring → Idle grey (`colors.idle`). Active ring → Indigo (already correct). Don't reuse the Active/teal hue for any in-progress state; keep teal exclusively for "the call is live" semantics. This restores the brief's intended warm/cool contrast between "alive" (teal) and "interactive/in-progress" (indigo/grey).

### Contacts → project header

- **The only interactive element on a project header is a 36dp tap target.** `ContactsRows.kt:90–105` wraps the call action in `IconButton(modifier = Modifier.size(36.dp))`. The brief mandates ≥48dp minimum on all interactive elements _"(Auto + driving use)"_. 36dp is well under that, and this is the surface where the user is most likely tapping while glancing — it is the worst possible place to under-size a hit target.
  - **Fix:** Lift the call IconButton to at least 48dp (preferably 56dp given the driving context). If layout density is a concern, also reconsider whether the entire project header row shouldn't be tappable — the brief originally treated the project as a 72dp Contact Row with a chevron; the implementation re-cast it as a section header with a small trailing action. The section-header approach is a defensible product call (visual hierarchy in a grouped list reads better), but the trailing action then has to be sized like a real call button, not a list-item icon.

### Setup → no back navigation

- **Settings cannot be exited back to Contacts.** `SetupScreen.kt:121–127` constructs the TopAppBar with title only — no leading `navigationIcon`. When the user reaches Setup from the Contacts gear icon (`onEditSettings`), the only ways back are (a) the system back gesture or (b) successfully connecting and tapping "Continue to contacts." For a user who opened Settings out of curiosity, with no intent to edit, the back gesture is the only path — and the brief explicitly relies on a visible nav affordance (_"leading back arrow (Ink) when navigation is possible"_).
  - **Fix:** Pass a `navigationIcon = { IconButton(onClick = onBack) { Icon(ArrowBack, "Back") } }` to the Setup TopAppBar, and thread an `onBack` parameter through `SetupScreen` so the Contacts→Settings nav has a real return path. The button should be visible whenever the screen was reached from somewhere else (i.e. always except on the initial onboarding entry, if you want to differentiate that case — but always-visible is also fine).

## HIGH

### In-Call screen → header zone

- **Header subtitle is a hardcoded label, not project context.** `InCallScreen.kt:155–159` renders the static string `"Voice call"` in bodyMedium under the session name. The brief specifies _"Session name in titleLarge (Ink), project name in bodyMedium (Ink secondary) below it, both center-aligned."_ Right now the user gets no visual confirmation of _which_ project they're calling — only the session name. On the In-Call surface this is the second-most-important piece of information after the agent state, and it's missing.
  - **Fix:** Wire the project label through the `CallViewModel`. Source it the same way `ContactsScreen` does (project basename for the row's `folderPath` via `PhoneAccountRules.rootSegmentOf` or the project meta). Replace the literal `"Voice call"` with that label. When no project context exists (somehow), omit the line entirely rather than padding it with a generic placeholder.

### In-Call screen → mute is interactive during connecting

- **Mute toggles before the call is Active.** `InCallScreen.kt:227–243` makes the Mute box clickable at all times. The brief is explicit: _"Dialing / Connecting: ... Mute is disabled (greyed out) until Active."_ The reason matters — tapping Mute pre-Active either does nothing useful (no audio path yet) or silently toggles a future state the user can't verify.
  - **Fix:** Disable the Mute control unless `state is CallState.Active`. Apply opacity 0.38 (the design system's disabled opacity) and skip the `clickable` while disabled. Same treatment for the Speaker button — it shouldn't take input while routing isn't yet live.

### In-Call screen → action buttons have no labels

- **Mute / Hang up / Speaker buttons are icon-only.** The brief specifies a `bodySmall` (Ink secondary) label _below_ each circular action button: _"Mute" / "Unmute" / "End call" / "Speaker"_. The implementation omits them. For a hands-free / glanced-at surface this is a real accessibility loss — icon-only controls are exactly what an eyes-off user can't disambiguate quickly. The brief also marked this under "Color is never the sole carrier of meaning" — currently mute state is communicated only via background color + icon swap, with no text affordance.
  - **Fix:** Add a 4–8dp gap and a centered `bodySmall` label below each action button. Labels follow the brief's content rules: "Mute"/"Unmute", "End call", "Speaker". This also helps Auto-style glance use even if the screen itself isn't rendered in Auto.

### Contacts → status pill doesn't actually collapse out of the layout

- **The "collapsed" pill still consumes a 48dp row above the list.** `StatusPill.kt:78–105` switches to a dot when collapsed, but renders that dot inside a `fillMaxWidth()` `Box` with `heightIn(min = 48.dp)` — sitting in the Column above the LazyColumn. The brief's intent was that on Connected the indicator _moves into the TopAppBar trailing area_ (_"collapses to a 6dp persistent dot in the app bar trailing area"_); instead it just shrinks but stays where it was, occupying the same vertical real estate.
  - **Fix:** Either move the collapsed dot into the TopAppBar `actions` slot (the original brief), or — if that's too invasive — collapse the entire pill row's height to 0 (animate the Column entry out) when Connected for >3s, and keep a tiny dot indicator inside the TopAppBar `actions` row alongside Refresh/Settings. The current "shrinks but doesn't move" state is the worst of both: still steals vertical space, no longer informative.

### Contacts → session row diverges substantially from brief

- **Session-row layout, typography, and icon treatment are all off-spec.** `ContactsRows.kt:163–202`:
  - Primary text uses `bodyLarge.copy(SemiBold)` instead of `titleMedium` (per brief spec); they're close but not the same role in the type system.
  - The cwd hint uses `fontStyle = Italic` — the brief's typography rules explicitly say _"No italic used anywhere."_
  - Leading icon is tinted `inkSecondary` at 18dp; brief specifies session icon at 24dp tinted **Indigo**. The current treatment makes the icon recede and signals "decoration" rather than "this row is a call action."
  - There's a third metadata line ("3 msgs · 2h ago") that wasn't in the brief — possibly a legitimate addition, but combined with the cwd line it makes rows denser than the brief's two-line target.
  - Row minHeight is 64dp; brief specified 72dp for the driving-use ergonomic floor.
  - **Fix:** Drop the italic on cwd (use a regular weight + Ink secondary; the indent + size already signals subordination). Re-tint the leading icon Indigo at 24dp to match the brief and to read as "this is a callable contact." Lift minHeight to 72dp. Consider whether the cwd line is high-signal enough to keep — if both cwd and metadata are kept, the row is doing more than the brief's two-line spec but at least make it read as deliberately three-line, not "two-line spec with a stray italic squeezed in."

### Setup → intro copy violates the brief's voice rules

- **The Cloudflare-Access instruction is one long technical paragraph.** `SetupScreen.kt:140–144` renders ~70 words of jargon-heavy explanation including literal backticks around `CF-Access-Client-Id` / `CF-Access-Client-Secret` (rendered as backticks, not styled). The brief's empty-state and intro voice rule is _"Two sentences max. First: state the situation. Second: say what to do."_ And: _"No marketing/jargon density on product surfaces."_ The current text serves the technical user but is heavier than the brief calls for.
  - **Fix:** Cut to two sentences: lead with the URL field's purpose, mention Cloudflare Access as optional in a single tail clause. Move the deep explanation (CF header names, "no WARP needed", "network-layer auth" nomenclature) into a "Learn more" link or inline help. Drop the backtick-quoted header names from the user-facing string — they're implementation detail. Suggested length: ≤ ~25 words for the intro, with the field labels carrying the rest of the information.

### In-Call screen → Speaker control is undocumented in the brief

- **Speaker button is a brief divergence that's reasonable but unaccounted for.** The brief enumerated exactly two action buttons (Mute, Hang up). The implementation correctly added a Speaker toggle — phones genuinely need it. But adding a third button breaks the brief's deliberate two-button composition and changes the visual rhythm of the action zone. As-is, it works, but the implementation should either be documented as a justified divergence (in a DR or in an updated brief) or rebalanced.
  - **Fix:** Update the brief (it's been deleted — restore + amend, or write a DR). For composition: with three buttons, the asymmetric Hang-Up-is-bigger (72dp vs 64dp) treatment becomes less effective because the Speaker button is far from Hang Up. Consider mirroring: Speaker + Mute as a left-side pair both at 64dp, Hang Up as the right-anchored 72dp primary destructive. Or center Hang Up and flank with Mute/Speaker symmetrically. Either way, the action zone needs to be a deliberate composition, not "we added a button on the left."

## MEDIUM

### Contacts → ProjectHeaderRow uses uppercase + letter-spacing

- **Section-header treatment is decorative.** `ContactsRows.kt:75–87` renders the project label as uppercase, letter-spaced labelMedium. The brief didn't explicitly ban uppercase here (it banned all-caps on _interactive_ labels), but the surrounding design DNA is "calm, achromatic restraint, no decorative type treatments." Linear/Signal/Aircall — the references — don't use uppercase section labels. The implementation reads slightly louder than the brief's overall voice.
  - **Fix:** Drop the `.uppercase()` and the `letterSpacing = 0.8.sp` override. Keep titleMedium-ish weight, Ink secondary color. The lower elevation tint (`surfacePlus` background) is doing the section-divider job already — text doesn't need to shout too.

### In-Call screen → unused `ContactRow` component

- **Dead component in the design system.** `components/ContactRow.kt` is fully implemented (148 lines) but no caller references it; `ContactsScreen` uses `ProjectHeaderRow` + `SessionListRow` from `contacts/ContactsRows.kt` instead. This isn't a design finding per se — it's a code-quality issue — but the _design_ artifact is that the component library now has two parallel row systems and no single source of truth for "what a Pimote list row looks like." Future contributors will pick one or the other inconsistently.
  - **Fix:** Either delete `ContactRow.kt` if the grouped layout is the canonical direction, or migrate `ContactsRows.kt` to reuse `ContactRow` as its base primitive. Don't leave two divergent implementations of the same idea in the design layer.

### In-Call screen → snackbar host has doubled padding

- **`PimoteSnackbarHost` applies 16dp padding twice.** `PimoteSnackbar.kt:35` puts 16dp on the host, then `:44` puts another 16dp on the Snackbar itself, giving 32dp insets from the screen edge. The brief specifies 16dp margin from screen edges; the implementation is double that.
  - **Fix:** Remove one of the two `.padding(16.dp)` calls. The intent reads as the outer host padding (so the inner Snackbar sits at its natural anchor); drop the inner one.

### Setup → status pill is below the Connect button

- **State indicator follows the action that triggers it.** `SetupScreen.kt:191–200` places the StatusPill after the Connect button. The brief's first-impression rule is _"the user's first question is always 'am I connected?' — answer it immediately, above the fold."_ On Setup, the question is more nuanced — the user is mid-flow — but burying the pill below the button means after tapping Connect, the user's eyes are above the line that's actually changing.
  - **Fix:** Move the StatusPill above the Connect button (between the field stack and the action). Failed/disconnected text remains visible while the user re-edits; on success it's right in the visual path between "what I typed" and "what to do next."

### Components → status pill mono font for label

- **`labelMedium` overridden with mono family.** `StatusPill.kt:81–82` synthesizes a custom style: `labelMedium.copy(fontFamily = mono)`. The brief specifies status strings _"use monospace type"_ and _"labelMedium ... Status badge text"_ — so combining them is correct intent. The execution synthesizes a non-canonical type token at the use site rather than introducing a `labelMono` token to the type scale.
  - **Fix:** Add a named style token (e.g. `labelMedium` already has tracking 0.4sp; introduce `monoLabel` to `PimoteTypography` that is monospace at the same metrics). Use the named token in the pill instead of an inline `.copy`. This is the kind of one-off "fix locally" pattern that erodes a type system over a few months.

### In-Call screen → "Voice call" placeholder when no session resolved

- **Fallback header text reads as a label, not a name.** `InCallScreen.kt:151–154` falls back to `"Pimote"` as the session-name text when `sessionName == null`. The brief's content rules called for the _first letter of the session/project name_ as the monogram in the avatar — when there's no name, the monogram defaults to "P", which is reasonable. But the header `"Pimote"` text is generic; when paired with the subtitle `"Voice call"` (see HIGH finding above), the entire header zone reads as boilerplate.
  - **Fix:** Once project context is wired into the header (per HIGH finding), the no-session-resolved case should display the _project name_ as the primary heading with no subtitle, rather than the brand name "Pimote." The brand belongs on the Contacts screen / app icon, not as in-call fallback chrome.

## LOW

### Components → ProjectHeaderRow showTopDivider asymmetry

- **The very first group has no top divider; subsequent groups do.** This is a deliberate decision in `ContactsScreen.kt:226` (`showTopDivider = row.groupIndex > 0`) and reads correctly. Mentioned only because it's worth keeping consistent if the empty-state path or a future "pinned" group reshuffles the order — the rule should be "no divider on the topmost rendered group," not "no divider on `groupIndex == 0`." Currently those are the same; in the future they may not be.
  - **Fix:** Treat as an observation. If the contacts list ever gains a "Recently called" or pinned section, re-derive `showTopDivider` from the row's position in the flattened list, not from `groupIndex`.

### Setup → Continue button variant

- **"Continue to contacts" uses Secondary variant.** The brief explicitly says: _"Used for: 'Continue to contacts,' 'Close,' settings actions"_ — so this is correct. Mentioned only to note that "Continue" as a secondary action means a user on a successful re-connect has no clear primary CTA on the screen. That's fine (the work is done), but worth knowing the screen's visual hierarchy goes flat post-success.
  - **Fix:** No action needed. Observation only.

## Surfaces examined with no notable findings

- **`PimoteOutlinedTextField`** — Faithful to brief: 12dp radius, label colors per state, helper text on error with sentence-case content rule, password visual transformation correctly used by Setup.
- **`PimoteButton`** — All four variants, palette, 12dp radius, 52dp height, 100ms press scale, 16% ink overlay all match the brief precisely. The fixed 16dp leading slot (for stable width across loading-toggle) is good production polish the brief didn't even ask for.
- **`StatusPill` expanded form** — Variant colors, label strings, monospace label, lower-sentence-case copy all match brief. The collapsed-state positioning is the issue (HIGH finding); the pill itself is otherwise clean.
- **`PimoteTheme` / `PimoteColors` / `PimoteSpacing` / `PimoteTypography` token system** — Palette hex values match the brief one-for-one. Type scale is exact, including the −0.5sp tracking on callDisplay. CompositionLocal token plumbing is solid.
- **`AvatarRing` motion** — Dashed-arc 1200ms rotation for connecting; 2400ms ease-in-out scale+alpha pulse for active; static stroke for ended states. All animation timings and easings match the brief. The only avatar finding is the ring _color_ assignment (CRITICAL above), not the motion.
- **`EmptyState` component** — Layout, type roles, icon at 48dp Idle tint, optional CTA via Secondary button, sync-icon rotation for Connecting state — all match brief.
- **In-app vector iconography** (`ic_call_*`, `ic_mic_*`, `ic_folder_outlined`, `ic_chat_bubble_outlined`, `ic_sync`, `ic_wifi_off`, `ic_signal_wifi_bad`, `ic_error_outline`) — Material outlined style, consistent stroke weight, no emoji. Faithful to the brief's "no emoji as functional icons" rule.

- **Android system notification (`CallStyle`)** posted by `CallForegroundService` — system-rendered, brief explicitly defers its design.
- **Android Auto contact picker** — system-rendered; brief's intent was ensuring the names registered via `PhoneAccountRules` / contacts sync are scannable. The contact-name generation logic looks faithful (`<root> <project>` in `ContactsSync`), but I didn't validate on an Auto device.
- **App / launcher icon** — brief flagged this as an open question; no brand assets yet exist.
- **System dialer search / Google Assistant voice-trigger surfaces** — system-rendered.
