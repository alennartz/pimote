# Manual Testing — voice-call-fullscreen-ui

## Smoke Suite

Focus hints scope this run to voice/calling. From `tools/manual-test/PLAN.md`:

- **Journey 8 — Voice call: bind, in-call, hangup.** The journey itself is rewritten by this topic: the in-call UI is no longer a thin banner but a full-screen calling-mode surface with three gestures. We exercise the modified shape end-to-end.

Journeys 1–7 are deliberately skipped per focus hint ("Skip surfaces unrelated to voice/calling"). They were unaffected by the diff (`+layout.svelte` mobile header — only the inline phone button block — was the sole non-voice file touched, and nothing in the journeys 1–7 paths references it).

## Topic-Specific Tests

For this topic, all checks slot underneath journey 8 — the topic _is_ the rewritten in-call surface. Each item maps to a focus-hint bullet:

- **(a) Entry from desktop StatusBar inline button.** `<CallButton sessionId variant="inline">` in `StatusBar.svelte` starts a call; calling-mode mounts.
- **(a) Entry from mobile SessionSettingsDialog row.** New `Voice call` row hosting `<CallButton variant="dialog-row">` between `Thinking` and `Context`. Click → calling-mode mounts.
- **(b1) Tap → mute toggle + mute-on / mute-off audio cue.** Cue pitch differs by direction (mute-on higher than mute-off, per `call-audio-cues.test.ts`).
- **(b2) Swipe up → hang up.** No audio cue (call audio drop is the cue).
- **(b3) Swipe down → abort.** Sends `AbortCommand` via `voiceCallStore.abortAgent()`; plays double-beep abort cue. Phase stays `connected`.
- **(c) Auto-return on call end.** `CallingMode` unmounts as soon as `voiceCallStore.state.phase` returns to `idle`; the normal chat surface for the same session re-renders. No flicker, no dead-call screen.
- **(d) Agent-state pulse.** Discrete state cycles `listening → thinking → speaking → listening` driven by `deriveAgentState` (priority: `remoteAudioLevel > threshold` ⇒ speaking, else `isStreaming` ⇒ thinking, else listening). Speaking visual additionally pulses with `remoteAudioLevel`.
- **(e1) `CallBanner.svelte` is fully gone.** No file, no imports, no mounts anywhere in `client/`.
- **(e2) Inline mobile-header phone button is fully gone.** The `{#if sessionRegistry.viewedSessionId} {@const callState ...}` block in `+layout.svelte` is deleted; only the side-effect import remains.

## Tools

- **Reused:** `agent-browser` skill (registered in `tools/manual-test/README.md` from the prior `voice-mode` re-test). Same setup recipe as `docs/manual-tests/voice-mode.md` — sandboxed `XDG_CONFIG_HOME` / `XDG_STATE_HOME`, dummy speechmux config (`/bin/true` binary, dead signal URL), `getUserMedia` shim via `agent-browser eval`.
- **Reused (static):** repo-local `npm run build`, `npx vitest run`, `npm run check` for code-path / structural assertions.
- **New:** none.
- **Improved:** none.

The dead-signal-URL recipe is well-suited to the topic: calling-mode mounts on `phase === 'binding' | 'connecting' | 'connected'`, so we can mount it without a real speechmux at the cost of never reaching `connected`. Anything that depends on `startedAt` (which only sets on `connected`) is verified by unit tests (`voice-call.svelte.test.ts → startedAt lifecycle`) plus a code-path inspection rather than live UI.

### Live agent-browser run notes

The `agent-browser`-driven smoke followed the `docs/manual-tests/voice-mode.md` recipe verbatim — sandboxed config, `/bin/true` speechmux binary, dead signal URL, `getUserMedia` shim — and confirmed:

- Clicking the desktop status-bar Call button replaces the chat surface with the calling-mode container (header at top with project label + duration `00:00` while in `binding`/`connecting`, transcript in the middle with `pointer-events: none` on its inner wrapper, gesture zone at bottom with the three permanent hint labels).
- Clicking `End call` (or invoking the swipe-up gesture path through `voiceCallStore.endCall()`) returns the screen to the normal chat surface for the same session — no banner residue, no dead-call confirmation screen, exactly the auto-return semantics the brainstorm specifies.
- No `CallBanner` strip ever appears, on either side of the call lifecycle.

Live-only items not directly reachable in the dead-signal-URL recipe (the call never reaches `connected`, so `startedAt`-driven duration ticking and analyser-driven `speaking` modulation cannot be observed end-to-end without real speechmux) are covered by the unit tests called out in the table above and have explicit code-path audits. The corresponding real-speechmux gap is the same pre-existing externally blocked item recorded in `docs/manual-tests/voice-mode.md → Open Issues`; it is _not_ specific to this topic.
